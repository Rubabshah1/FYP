\subsection{Classes and Services}

\subsubsection{Application-Layer Services}

\textbf{APIGateway}

\begin{itemize}\setlength{\itemsep}{0pt}
\item \textbf{Purpose}: Authentication, rate-limiting, routing to backend services, request validation.
\item \textbf{Attributes}: tokenStore (JWT secret), rateLimitConfig.
\item \textbf{Methods}: authenticate(req), authorize(req), route(req), rateLimit(req).
\item \textbf{Relationships}: Front door for WebClient \& WhatsAppClient $\rightarrow$ ChatEngine, AdminService.
\end{itemize}

\textbf{ChatEngine}

\begin{itemize}\setlength{\itemsep}{0pt}
\item \textbf{Purpose}: Orchestrates conversational processing: receives message, manages session, calls intent classifier, multimodal processor, RAG engine, and returns responses.
\item \textbf{Attributes}: sessionStore (Redis), kbClient, classifierClient, ragClient, escalationPolicy.
\item \textbf{Methods}:
\begin{itemize}\setlength{\itemsep}{0pt}
\item processIncoming(message: Message) -- main entry.
\item classifyIntent(text) $\rightarrow$ IntentResult (intent, confidence).
\item handleMultimodal(attachments) $\rightarrow$ structured artifacts.
\item generateResponse(context) $\rightarrow$ final message (via RAGEngine + prompt).
\item escalate(session, reason) $\rightarrow$ create Ticket and route to HumanDashboard.
\end{itemize}
\item \textbf{Relationships}: calls IntentClassifier, MultimodalProcessor, RAGEngine, DataStore.
\end{itemize}

\textbf{AdminService}

\begin{itemize}\setlength{\itemsep}{0pt}
\item \textbf{Purpose}: CRUD for knowledge base, guardrails, role management.
\item \textbf{Attributes}: KBRepository, userManager.
\item \textbf{Methods}: createArticle(), updateArticle(), publishKBVersion(), setGuardrails().
\item \textbf{Relationships}: updates KnowledgeArticle entries used by RAGEngine.
\end{itemize}

\textbf{HumanDashboard (HumanAgent)}

\begin{itemize}\setlength{\itemsep}{0pt}
\item \textbf{Purpose}: Staff-facing UI + API for taking over escalated tickets, replying to users, updating KB entries.
\item \textbf{Attributes}: ticketQueue, websocketConnections.
\item \textbf{Methods}: claimTicket(ticketId), postReply(ticketId, message), updateKBFromInteraction(...).
\item \textbf{Relationships}: Ticket $\leftrightarrow$ Agent $\leftrightarrow$ Session.
\end{itemize}

\textbf{AnalyticsService}

\begin{itemize}\setlength{\itemsep}{0pt}
\item \textbf{Purpose}: Log metrics, build KPIs (accuracy, latency, escalation rates).
\item \textbf{Attributes}: metricsStore, logDB.
\item \textbf{Methods}: logQuery(...), getKPI(start,end), alertAnomalies().
\item \textbf{Relationships}: receives events from ChatEngine and AdminService.
\end{itemize}

\subsubsection{AI/ML classes/services}

\textbf{IntentClassifier}

\begin{itemize}\setlength{\itemsep}{0pt}
\item \textbf{Purpose}: Predicts domain \& intent from textual input; returns confidence score.

\begin{itemize}\setlength{\itemsep}{0pt}

\item \textbf{Attributes}: visionModel, ocrEngine (Tesseract or cloud), imagePreproc.
\item \textbf{Methods}: processImage(img) $\rightarrow$ \{text, tags, doc\_type, confidence\}.

\begin{itemize}\setlength{\itemsep}{0pt}

\item \textbf{Methods}: retrieve(query,k), buildPrompt(context, docs), generateAnswer(prompt) $\rightarrow$ answer, sources.

\section{Data Design}

This section presents the normalized relational model, cardinalities, and the core relational schema. The model is designed to conform to third normal form (3NF).

\subsection{Entities, Attributes, Primary/Foreign Keys, and Cardinalities}

The primary relationships between entities are summarized as follows:

\begin{itemize}\setlength{\itemsep}{0pt}
\item \textbf{User (1) -- (0..n) Session}: One user can have many sessions. This design allows tracking the number of interactions and building user-based context for future queries.

\item \textbf{Session (1) -- (0..n) Message}: A session contains many messages.

\item \textbf{Session (1) -- (0..1) Ticket}: A session may create one open ticket.

\item \textbf{Ticket (0..n) -- (0..1) Agent}: A ticket may be assigned to one agent; agents handle many tickets.

\item \textbf{KnowledgeArticle (0..n) -- (0..n) Message (reference)}: Messages (bot replies) may reference multiple knowledge articles and vice versa, implemented through a junction table.

\item \textbf{BloodBank (1) -- (0..n) BloodStock}: Each blood bank has multiple blood stock entries (one entry per blood group).
\end{itemize}

\subsection{Indexes and Search}

The indexing strategy includes the following components:

\begin{itemize}\setlength{\itemsep}{0pt}
\item Full-text indexing using either Elasticsearch or PostgreSQL tsvector on the \texttt{knowledge\_articles.content} and \texttt{knowledge\_articles.title} fields.

\item Vector embeddings are stored in a dedicated vector store (FAISS, Milvus, or Elasticsearch vector index), keyed by \texttt{doc\_id}. Vector metadata is maintained in a lightweight \texttt{kb\_index} table that establishes the mapping between \texttt{doc\_id} and \texttt{vector\_id}.
\end{itemize}

\subsection{Normalization and Cardinalities Rationale}

The schema design incorporates the following principles:

\begin{itemize}\setlength{\itemsep}{0pt}
\item The schema explicitly separates knowledge articles (the canonical knowledge base) from message-level references. This separation avoids data duplication while supporting many-to-many referencing relationships, maintaining third normal form (3NF).

\item The \texttt{sessions} table stores transient conversation context, while messages are treated as transactional records that cascade upon session deletion.

\item The \texttt{blood\_stock} table includes a unique constraint on the composite key (\texttt{bank\_id}, \texttt{blood\_group}). This constraint prevents the creation of duplicate rows and simplifies the process of updating stock levels.

\item The \texttt{message\_article\_refs} junction table preserves provenance information, explicitly recording which knowledge base articles supported a specific bot response.
\end{itemize}

\subsection{Technical Details}

\subsubsection{Data Pipeline}

The system implements an end-to-end data pipeline for processing both unstructured and structured data sources:

\begin{enumerate}
    \item \textbf{Web Scraping}: Periodic crawlers extract content from designated websites, parse structured fields, and store raw output with checksums in a staging area.
    \item \textbf{Document Repository}: Verified content is persisted in Google Drive with comprehensive metadata including domain classification, source URL, and checksums.
    \item \textbf{Structured Data Management}: Blood bank operational data is maintained in a relational database with change tracking and audit logging.
    \item \textbf{Preprocessing}: Text cleaning, language detection, Roman-Urdu normalization, semantic-aware chunking (300–500 tokens), and image OCR processing.
    \item \textbf{Embedding and Indexing}: Document chunks and structured data summaries are converted to embeddings and stored in a vector database with associated metadata.
    \item \textbf{RAG Retrieval and Generation}: Query embedding followed by vector search, result reranking, prompt construction with retrieved context, and LLM generation.
\end{enumerate}

\subsubsection{Retrieval Strategy}

The retrieval process combines vector similarity search with metadata filtering and reranking:

\begin{itemize}\setlength{\itemsep}{0pt}
    \item Initial retrieval of top-k candidate documents using vector similarity
    \item For blood availability queries, simultaneous retrieval from structured BloodStock database
    \item Reranking based on multiple signals: cosine similarity, document recency, domain relevance, and exact match criteria (city, blood group)
    \item Selection of top-n documents for prompt construction
\end{itemize}

\subsubsection{Prompt Construction and Generation}

Responses are generated using carefully structured prompts that explicitly constrain the LLM to use only provided source documents:

\begin{quote}
SYSTEM: You are a domain-expert assistant. Use only the documents listed in SOURCES.

SOURCES: [numbered document excerpts with identifiers]

TASK: Provide a concise, evidence-based answer. For each factual claim include the relevant source identifier. If no verified information is available in the provided sources, explicitly state "No verified information is available."
\end{quote}

\subsubsection{Image Processing Pipeline}

All uploaded and scraped images undergo comprehensive processing:

\begin{itemize}\setlength{\itemsep}{0pt}
    \item Preprocessing including resizing, denoising, deskewing, and grayscale conversion
    \item OCR processing with language hinting for English, Urdu, and Roman-Urdu
    \item Post-OCR named entity recognition and structured extraction for key entities
    \item Confidence-based quality control with fallback handling for low-confidence results
\end{itemize}

\subsubsection{Language Handling}

The system addresses multilingual requirements through:

\begin{itemize}\setlength{\itemsep}{0pt}
    \item Language detection for input classification
    \item Roman-Urdu normalization using mapping tables and transliteration heuristics
    \item Multilingual embedding models capable of representing semantic similarity across English, Urdu, and Roman-Urdu content
\end{itemize}

\subsubsection{Monitoring and Quality Control}

The system includes comprehensive monitoring and validation mechanisms:

\begin{itemize}\setlength{\itemsep}{0pt}
    \item Post-generation source coverage verification to ensure response content is supported by retrieved documents
    \item Confidence scoring based on retrieval quality, source coverage, and supporting evidence
    \item Escalation mechanisms for low-confidence responses and explicit user requests for human intervention
    \item Metrics collection including query latency, knowledge base hit rate, escalation rate, and retrieval precision
\end{itemize}

\subsubsection{Blood Availability Query Processing}

Blood availability queries follow a specialized processing path that combines structured database lookup with document retrieval:

\begin{enumerate}
    \item Intent classification identifies blood availability requirement
    \item Entity extraction determines required blood group and geographic constraints
    \item Direct structured query against BloodStock table with appropriate filters
    \item Parallel retrieval of supporting knowledge documents (eligibility criteria, procedural information)
    \item Response construction including verified stock information with explicit source attribution
    \item Validation ensuring all numerical claims correspond to authoritative database records
\end{enumerate}

\subsubsection{Failure Modes and Mitigations}

\begin{itemize}\setlength{\itemsep}{0pt}
    \item \textbf{Stale stock information}: Explicit timestamp presentation and recommendation that recipients verify availability by contacting blood banks directly
    \item \textbf{OCR processing failures}: Confidence-based detection with user prompts for image reupload
    \item \textbf{Retrieval failures}: Fallback to keyword search or predefined FAQ responses
    \item \textbf{Hallucination prevention}: Strict prompt grounding, source verification, and low-temperature generation parameters
\end{itemize}

\subsection{Summary}

The described system architecture implements a comprehensive Retrieval-Augmented Generation framework specifically tailored for blood donation and healthcare support applications. The architecture combines structured operational data management with extensive knowledge base indexing, sophisticated retrieval and reranking mechanisms, and robust multimodal processing capabilities. The design emphasizes factual grounding through explicit source attribution, confidence-based quality control, and comprehensive escalation pathways to ensure reliable, verifiable responses across all operational scenarios.
